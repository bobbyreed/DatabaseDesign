Historical Context üìú

From SEQUEL to SQL

SEQUEL to SQL

SEQUEL (Structured English Query Language) was developed for IBM's System R by Don Chamberlin and Raymond Boyce. Due to a trademark conflict, it was renamed SQL (Structured Query Language). SQL was deliberately designed to be more user-friendly than relational calculus while retaining its theoretical power‚Äîmaking database queries accessible to non-mathematicians.

Early Standardization (1986/1987)

SQL's rapid adoption led to standardization by ANSI in 1986 and ISO in 1987, cementing its status as the "intergalactic standard" for relational databases. This standardization ensured SQL would work across different database systems, making it the universal language for data management.

Key Insight: SQL's success came from balancing theoretical rigor with practical usability‚Äîit's powerful enough for complex queries yet simple enough for everyday use.

History of SQL

The Birth of a Standard

  • 1974: D. Chamberlin (IBM) creates SEQUEL - "Structured English Query Language"
  • 1976: SEQUEL/2 defined, renamed to SQL for legal reasons
  • Late 1970s: ORACLE appears as first commercial RDBMS based on SQL
  • 1987: ANSI and ISO publish initial SQL standard
  • 1992: SQL2/SQL-92 - First major revision
  • 1999: SQL:1999 - Object-oriented features
  • 2003-2019: Continuous evolution (SQL:2003, 2008, 2011, 2016, 2019)

Note: Still pronounced "see-quel" by many, though official pronunciation is "S-Q-L"

Why SQL Matters

Industry Standard and Beyond

  • Part of major application architectures (IBM's Systems Application Architecture)
  • Strategic choice of influential organizations (X/OPEN)
  • Federal Information Processing Standard (FIPS)
  • Required for all database sales to U.S. Government
  • Used in other standards (ISO's IRDS, Remote Data Access)
  • De facto standard for relational databases

The SELECT Statement

Foundation of SQL Queries

SELECT [DISTINCT | ALL] { * | [columnExpression [AS newName]] [, ...] }
FROM TableName [alias] [, ...]
[WHERE condition]
[GROUP BY columnList] [HAVING condition]
[ORDER BY columnList]

Key Points:

  • Order of clauses cannot be changed
  • Only SELECT and FROM are mandatory
  • Keywords are RESERVED WORDS

Aggregate Functions

Statistical Operations

Available Functions

  • COUNT(*) - Number of rows
  • COUNT(column) - Non-null values
  • SUM(column) - Total of values
  • AVG(column) - Average value
  • MIN(column) - Minimum value
  • MAX(column) - Maximum value

Example

SELECT COUNT(*) AS staff_count,
    AVG(salary) AS avg_salary,
    MAX(salary) AS max_salary
FROM Staff;

Important: Aggregate functions can only be used in SELECT list and HAVING clause

GROUP BY Clause

Grouping for Subtotals

Example: Find number of staff in each branch and their total salaries

SELECT branchNo,
    COUNT(staffNo) AS myCount,
    SUM(salary) AS mySum
FROM Staff
GROUP BY branchNo
ORDER BY branchNo;
branchNomyCountmySum
B003354000.00
B005239000.00
B00719000.00

Rule: All columns in SELECT must either be in GROUP BY or be aggregate functions

HAVING Clause

Filtering Grouped Results

Example: For branches with more than 1 staff member, show count and sum of salaries

SELECT branchNo,
    COUNT(staffNo) AS myCount,
    SUM(salary) AS mySum
FROM Staff
GROUP BY branchNo
HAVING COUNT(staffNo) > 1
ORDER BY branchNo;

WHERE vs HAVING

  • WHERE: Filters individual rows
  • HAVING: Filters groups
  • WHERE comes before GROUP BY
  • HAVING comes after GROUP BY

Pattern Matching with LIKE

Flexible String Searching

Special Pattern Symbols

  • % - Zero or more characters
  • _ - Exactly one character

Examples

-- Find all addresses in Glasgow
SELECT * FROM Branch
WHERE city LIKE '%Glasgow%';

-- Find staff with 'o' as second letter of first name
SELECT * FROM Staff
WHERE fName LIKE '_o%';

-- Find properties on streets ending with 'Rd'
SELECT * FROM PropertyForRent
WHERE street LIKE '%Rd';

Writing Good SQL Commands

Best Practices by Bobby Reed

‚úÖ DO

  • Use consistent capitalization for keywords
  • Start each clause on a new line
  • Indent subparts of clauses
  • Use meaningful aliases
  • Comment complex queries
  • Test queries incrementally

‚ùå DON'T

  • Write everything on one line
  • Use SELECT * in production
  • Forget to handle NULLs
  • Ignore performance implications
  • Mix different formatting styles
  • Use reserved words as names
-- Good formatting example
SELECT s.staffNo, s.fName, s.lName,
       COUNT(p.propertyNo) AS properties_managed
FROM Staff s
    LEFT JOIN PropertyForRent p ON s.staffNo = p.staffNo
WHERE s.position = 'Manager'
GROUP BY s.staffNo, s.fName, s.lName
HAVING COUNT(p.propertyNo) > 0
ORDER BY properties_managed DESC;

Common SQL Pitfalls

Learn from Others' Mistakes

1. Forgetting GROUP BY

-- ‚ùå WRONG: Missing GROUP BY
SELECT branchNo, COUNT(*)
FROM Staff;

-- ‚úÖ CORRECT:
SELECT branchNo, COUNT(*)
FROM Staff
GROUP BY branchNo;

2. Wrong Clause Order

-- ‚ùå WRONG: ORDER BY before WHERE
SELECT * FROM Staff
ORDER BY salary
WHERE salary > 10000;

-- ‚úÖ CORRECT:
SELECT * FROM Staff
WHERE salary > 10000
ORDER BY salary;

Performance Considerations

Writing Efficient Queries

üöÄ Quick Performance Tips

  • Be Specific: Select only columns you need, not SELECT *
  • Filter Early: Use WHERE to reduce data before grouping
  • Index Awareness: Filters on indexed columns are faster
  • Limit Results: Use LIMIT/TOP when testing queries
  • Avoid Wildcards at Start: LIKE '%text' is slower than LIKE 'text%'

Example: Improving Query Performance

-- Slower: Gets all columns, then filters
SELECT *
FROM PropertyForRent
WHERE city LIKE '%Glasgow%';

-- Faster: Specific columns, exact match if possible
SELECT propertyNo, street, rent
FROM PropertyForRent
WHERE city = 'Glasgow';

Exercise 1: Basic Select & Filter

Individual Exercise (20 minutes)

Your Task:

Using the Student/Course schema below, write and execute five basic SELECT queries, each using a different WHERE condition.

Schema:

Student(studentID, firstName, lastName, email, major, GPA, enrollmentDate)
Course(courseID, courseName, department, credits, instructorName)
Enrollment(studentID, courseID, semester, grade, enrollDate)

Write queries for:

  1. Equality: Find all students with major = 'Computer Science'
  2. Range: Find all students with GPA between 3.0 and 4.0
  3. Comparison: Find all courses with credits > 3
  4. LIKE: Find all students whose email ends with '@okcu.edu'
  5. Combined: Find CS majors with GPA >= 3.5

Understanding: SELECT & Filter Queries

Solutions:

-- Query 1: Equality
SELECT * FROM Student
WHERE major = 'Computer Science';

-- Query 2: Range (using BETWEEN)
SELECT studentID, firstName, lastName, GPA
FROM Student
WHERE GPA BETWEEN 3.0 AND 4.0;

-- Query 3: Comparison
SELECT courseID, courseName, credits
FROM Course
WHERE credits > 3;

-- Query 4: Pattern matching with LIKE
SELECT firstName, lastName, email
FROM Student
WHERE email LIKE '%@okcu.edu';

-- Query 5: Combined conditions
SELECT studentID, firstName, lastName, major, GPA
FROM Student
WHERE major = 'Computer Science'
  AND GPA >= 3.5;

Understanding: WHERE Condition Types

Key Insights from Exercise 1:

Condition Type Operator Use Case
Equality = Exact match (major = 'Computer Science')
Range BETWEEN...AND Values within a range (GPA 3.0-4.0)
Comparison >, <, >=, <=, != Relative comparisons (credits > 3)
Pattern Match LIKE with % and _ Flexible string matching
Combined AND, OR, NOT Multiple conditions together

Performance Tip:

Pattern matching with LIKE '%text' (wildcard at start) is slower than 'text%' because it can't use indexes efficiently.

Exercise 2: Simple Data Manipulation (DML)

Individual Exercise (20 minutes)

‚ö†Ô∏è Important:

DML statements modify data. Always double-check your WHERE clauses!

Your Task:

Write the appropriate INSERT, UPDATE, and DELETE statements for these three scenarios:

Scenario 1: INSERT

A new student has enrolled:

  • studentID: 'S12345'
  • firstName: 'Maria', lastName: 'Garcia'
  • email: 'maria.garcia@okcu.edu'
  • major: 'Data Science', GPA: 3.8
  • enrollmentDate: '2024-01-15'

Scenario 2: UPDATE

Student 'S10001' has changed their major from 'Biology' to 'Biochemistry' and their new GPA is 3.65.

Scenario 3: DELETE

Remove all enrollment records for semester 'Fall2020' (old data cleanup).

Understanding: DML Operations

Solutions:

-- Scenario 1: INSERT new student
INSERT INTO Student (studentID, firstName, lastName, email, major, GPA, enrollmentDate)
VALUES ('S12345', 'Maria', 'Garcia', 'maria.garcia@okcu.edu',
        'Data Science', 3.8, '2024-01-15');

-- Scenario 2: UPDATE existing student
UPDATE Student
SET major = 'Biochemistry',
    GPA = 3.65
WHERE studentID = 'S10001';

-- Scenario 3: DELETE old enrollment records
DELETE FROM Enrollment
WHERE semester = 'Fall2020';

‚ö†Ô∏è Critical Safety Rules:

  • Always use WHERE: UPDATE and DELETE without WHERE affect ALL rows!
  • Test with SELECT first: Run SELECT with same WHERE to verify which rows will be affected
  • Use transactions: Wrap modifications in BEGIN/COMMIT for rollback capability

Understanding: DML Best Practices

Testing DML Before Execution:

-- Before UPDATE, test with SELECT
SELECT * FROM Student
WHERE studentID = 'S10001';
-- Verify this returns the correct student!

-- Then execute UPDATE
UPDATE Student
SET major = 'Biochemistry', GPA = 3.65
WHERE studentID = 'S10001';

-- Verify the change
SELECT * FROM Student
WHERE studentID = 'S10001';

Common DML Mistakes to Avoid:

  • ‚ùå Forgetting WHERE clause (affects all rows!)
  • ‚ùå Using wrong operator (= vs IN for multiple values)
  • ‚ùå Violating foreign key constraints
  • ‚ùå Not checking affected row count after execution

Understanding Transactions

Transaction Template:

BEGIN TRANSACTION;

    -- Your INSERT statement
    INSERT INTO Student (...) VALUES (...);

    -- Your UPDATE statement
    UPDATE Student SET ... WHERE ...;

    -- Your DELETE statement
    DELETE FROM Enrollment WHERE ...;

-- If all successful:
COMMIT;

-- If error occurs or you want to undo:
-- ROLLBACK;

ACID Properties of Transactions:

  • Atomicity: All or nothing - either all statements succeed or none do
  • Consistency: Database moves from one valid state to another
  • Isolation: Concurrent transactions don't interfere
  • Durability: Committed changes are permanent
