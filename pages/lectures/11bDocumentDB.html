What is a Document Database?

Definition: A type of NoSQL database that stores and queries data as JSON-like documents

Primary Example: MongoDB - the "M" in the MERN stack

  • üìÑ Stores semi-structured data in BSON (Binary JSON) format
  • üîß Flexible, evolving schemas - no migrations needed
  • üå≥ Hierarchical document structure with nested objects
  • üöÄ Designed for modern web and mobile applications
  • ‚ö° Horizontal scaling with sharding and replication
MERN Stack: MongoDB + Express.js + React + Node.js - a complete JavaScript stack for modern web development

Relational vs Document Databases

üóÉÔ∏è Relational (PostgreSQL)

  • Predefined schema
  • Tables with rows & columns
  • Data spread across tables
  • Requires JOINs
  • ACID transactions
  • SQL queries
  • Vertical scaling

üìÑ Document (MongoDB)

  • Flexible schema
  • Collections of documents
  • Self-contained documents
  • $lookup instead of JOINs
  • Eventual consistency*
  • JavaScript-like queries
  • Horizontal scaling (sharding)
Note: MongoDB now supports multi-document ACID transactions (since v4.0)

Document Database CRUD Operations

Core Operations

  • ‚ûï Create: Insert new documents with unique keys
  • üîç Read: Query by keys or field values
  • ‚úèÔ∏è Update: Modify entire document or specific fields
  • üóëÔ∏è Delete: Remove documents by key or query
Key Point: Each document has a unique identifier (key) for fast retrieval

MongoDB CRUD Examples

// CREATE - Insert a movie
db.movies.insertOne({
  title: "Inception",
  year: 2010,
  rating: 8.8
})

// READ - Find movies from 2010 or later
db.movies.find({
  year: {$gte: 2010}
})

// UPDATE - Modify rating
db.movies.updateOne(
  {title: "Inception"},
  {$set: {rating: 9.0}}
)

// DELETE - Remove a movie
db.movies.deleteOne({
  title: "Inception"
})

Document vs Key-Value Stores

üîë Key-Value Store

  • Simple key‚Üívalue pairs
  • Value is opaque (black box)
  • No query on value content
  • Returns entire value
  • Example: Redis, Memcached

üìÑ Document Store

  • Key‚Üístructured document
  • Understands document structure
  • Query nested fields
  • Returns partial documents
  • Example: MongoDB, CouchDB
Example: Query "book.price" returns just 10, not entire book object

Schema Migration Decision Tree

Relational ‚Üí Document Migration Rules:

1:1 Relationship ‚Üí Embed smaller entity
1:N Relationship ‚Üí Consider:
  - Few (< 100) ‚Üí Embed
  - Many (> 100) ‚Üí Reference
  - Growing unbounded ‚Üí Reference
  
M:N Relationship ‚Üí Usually Reference

Frequently accessed together ‚Üí Embed
Independently accessed ‚Üí Reference
Need transactions ‚Üí Reference
Remember: Optimize for your most common queries!

Performance Trade-offs

Aspect Relational Document
Read Performance Multiple JOINs slow Single document fast
Write Performance Normalized = fast Denormalized = slower
Consistency ACID guaranteed Eventual consistency
Scaling Vertical (bigger server) Horizontal (more servers)
Storage Efficient (normalized) More space (denormalized)

Choosing the Right Database

Use Relational (PostgreSQL) When:

  • ‚úÖ ACID transactions critical
  • ‚úÖ Complex relationships & referential integrity
  • ‚úÖ Structured, stable schema
  • ‚úÖ Reporting & analytics
  • ‚úÖ Financial/banking systems

Use MongoDB When:

  • ‚úÖ Rapid development (MERN stack)
  • ‚úÖ Evolving/flexible schemas
  • ‚úÖ Hierarchical/nested data
  • ‚úÖ Content management systems
  • ‚úÖ Real-time web/mobile apps

Today's Key Takeaways üéì

Document Database Fundamentals

  • ‚úÖ Flexible schemas with JSON/BSON documents
  • ‚úÖ No JOINs needed - data locality and denormalization
  • ‚úÖ Collections store documents (like tables store rows)
  • ‚úÖ Documents in same collection can have different structures

MongoDB Design Patterns

  • ‚úÖ Embedding: Nest related data (1:1, 1:few) - single query, 16MB limit
  • ‚úÖ Referencing: Separate collections (1:many, M:N) - no duplication, multiple queries
  • ‚úÖ Choose based on query patterns and data access frequency
  • ‚úÖ MongoDB CRUD operations: insertOne/Many, find, updateOne/Many, deleteOne/Many

MERN Stack Integration

  • ‚úÖ MongoDB serves as the database layer in MERN (MongoDB, Express, React, Node)
  • ‚úÖ Mongoose ODM provides schema validation and easier data modeling
  • ‚úÖ Horizontal scaling for modern, high-traffic applications

Hands-On: MongoDB Atlas Setup üöÄ

20 minutes

What We'll Do:

  1. Create a free MongoDB Atlas cluster
  2. Configure database user and network access
  3. Create the "taskmanager" database and "tasks" collection

Why Atlas? MongoDB Atlas is the official cloud database service - fully managed, free tier available, and production-ready. This is what you'll use in the MERN stack!

Step 1: Create MongoDB Atlas Account

1.1 Sign Up for MongoDB Atlas

  • Go to https://www.mongodb.com/cloud/atlas/register
  • Sign up with email or Google account
  • Complete email verification

1.2 Choose Free Tier

  • Select "Shared" cluster (M0 - Free)
  • Choose cloud provider: AWS
  • Choose region: Closest to you (e.g., us-east-1)
  • Cluster name: Cluster0 (default is fine)
  • Click "Create Cluster"
‚úì CHECKPOINT: Cluster creation begins (takes 3-5 minutes)

Step 2: Configure Database Security

2.1 Create Database User

While cluster is deploying, set up authentication:

  • In "Security Quickstart", create a database user
  • Username: dbUser (or your choice)
  • Password: Generate a secure password
  • IMPORTANT: Copy and save this password securely!
  • Click "Create User"

2.2 Configure Network Access

  • Click "Add My Current IP Address"
  • For deployment later, also add 0.0.0.0/0 (allows all IPs)
  • Description: "Allow all IPs for cloud deployment"
  • Click "Add IP Address"
Security Note: 0.0.0.0/0 allows any IP. This is acceptable for learning, but in production, restrict to your server's IP addresses.

Step 3: Get Connection String

3.1 Navigate to Connect

  • Wait for cluster to finish deploying (green status)
  • Click "Connect" button on your cluster
  • Select "Connect your application"
  • Driver: Node.js, Version: 5.5 or later

3.2 Copy Connection String

mongodb+srv://dbUser:<password>@cluster0.xxxxx.mongodb.net/?retryWrites=true&w=majority
  • Copy this connection string
  • Replace <password> with your actual database password
  • Save this modified string - you'll need it in the next lecture!
‚úì CHECKPOINT: You have a working connection string with password filled in

Step 4: Create Database & Collection

4.1 Browse Collections

  • From cluster overview, click "Browse Collections"
  • Click "Add My Own Data"

4.2 Create Database and Collection

  • Database Name: taskmanager
  • Collection Name: tasks
  • Click "Create"

4.3 Insert Sample Document (Optional but Recommended)

  • Click on the tasks collection
  • Click "Insert Document"
  • Paste the following:
{
  "title": "Learn MongoDB",
