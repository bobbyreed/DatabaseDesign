Historical Context ğŸ“œ

Foundations of Modern Database Design

E.F. Codd's 1970 Paper

Edgar F. Codd published his seminal paper, "A Relational Model of Data for Large Shared Data Banks" in 1970. This groundbreaking work fundamentally changed how data was logically structured and stored, introducing the revolutionary concept of separating logical organization from physical storage.

Peter Chen and the ER Model (1976)

Peter Chen created the Entity-Relationship Model (ERM) in 1976, providing the first widely accepted graphical tool for conceptual database design. This model became the foundation for modern database design methodologies and is still extensively used today.

Key Insight: These two innovationsâ€”Codd's relational model and Chen's ER diagramsâ€”transformed database design from an ad-hoc practice into a principled discipline.

Limitations of File-Based Approach (Part 1)

1. Separation and Isolation of Data

  • Each program maintains its own set of data
  • Users of one program may be unaware of potentially useful data held by other programs

2. Duplication of Data

  • Same data is held by different programs
  • Wasted space and potentially different values and/or different formats for the same item

Limitations of File-Based Approach (Part 2)

3. Data Dependence

File structure is defined in the program code

4. Incompatible File Formats

Programs are written in different languages, and so cannot easily access each other's files

5. Fixed Queries/Proliferation of Application Programs

  • Programs are written to satisfy particular functions
  • Any new requirement needs a new program

The Database Approach

Why Databases Emerged:

  • Definition of data was embedded in application programs, rather than being stored separately and independently
  • No control over access and manipulation of data beyond that imposed by application programs

The Result:

The birth of the Database and Database Management System (DBMS)

What is a Database?

Definition:

A shared collection of logically related data (and a description of this data), designed to meet the information needs of an organization.

Key Components:

  • System catalog (metadata): Provides description of data to enable program-data independence
  • Logically related data: Comprises entities, attributes, and relationships of an organization's information
Remember: A database is not just data - it's organized, described, and purposeful!

Database Management System (DBMS)

Definition:

A software system that enables users to define, create, maintain, and control access to the database.

Database Application Program:

A computer program that interacts with the database by issuing an appropriate request (SQL statement) to the DBMS.

Database Languages

Data Definition Language (DDL)

  • Permits specification of data types, structures and any data constraints
  • All specifications are stored in the database

Data Manipulation Language (DML)

  • General enquiry facility (query language) of the data
  • Most common: SQL (Structured Query Language)

Roles in the Database Environment

1. Data Administrator (DA)

Responsible for the management of data resources including database planning, development and maintenance of standards, policies and procedures

2. Database Administrator (DBA)

Responsible for the physical realization of the database, including physical database design and implementation, security and integrity control

3. Database Designers

  • Logical: Identifies the data, relationships, and constraints
  • Physical: Decides how the logical database design is physically realized

More Database Roles

4. Application Programmers

Develop application programs that provide the required functionality for the end-users

5. End Users

  • Naive Users: Unaware of the DBMS, access database through specific applications
  • Sophisticated Users: Familiar with the structure of the database and the facilities offered by the DBMS

JOIN Query: Solution

Combining Multiple Tables

Complete JOIN Query:

-- Show member name, movie title, and rent date for unreturned movies
SELECT m.name, b.title, br.rental_date
FROM Rentals br
JOIN Members m ON br.member_id = m.member_id
JOIN Movies b ON br.movie_id = b.movie_id
WHERE br.return_date IS NULL;

How It Works:

  1. Start with Rentals table - This is our "anchor" (has foreign keys to both other tables)
  2. JOIN Members - Match br.member_id with m.member_id to get member names
  3. JOIN Movies - Match br.movie_id with b.movie_id to get movie titles
  4. WHERE clause - Filter to only show movies that haven't been returned yet

Key Concepts:

  • JOIN: Combines rows from two tables based on a related column
  • Table Aliases: br, m, b make queries shorter and readable
  • Foreign Key â†’ Primary Key: Always match the foreign key to its corresponding primary key

Understanding: Entities and Relationships ğŸ”

What are Entities?

Entities are the "things" or objects in the real world that we want to store information about in our database.

  • They become tables in a relational database
  • Each entity should represent a distinct concept
  • Examples: Student, Course, Professor, Department

What are Relationships?

Relationships describe how entities are connected to each other.

Relationship Type Description Example
One-to-One (1:1) One entity instance relates to exactly one instance of another Student â†’ Student Profile
One-to-Many (1:N) One entity instance relates to many instances of another Professor â†’ Courses Taught
Many-to-Many (M:N) Many instances of one entity relate to many of another Students â†” Courses (via Enrollment)
Key Concept: Many-to-many relationships typically require a "junction table" (like Enrollment) to properly represent them in a relational database.

Common Design Flaws in Schema A:

  • Multivalued Attributes: genres, albums_list, all_songs_list stored as comma-separated values
    • âŒ Makes querying difficult: "Find all artists in genre X"
  • Data Duplication: artist_name, album_title, release_year duplicated in Song table
    • âŒ Inconsistency risk if artist name changes
  • Missing Relationships: No explicit Album entity
    • âŒ Can't efficiently manage album-level data

Common Design Flaws in Schema B:

  • Extreme Data Redundancy: artist_bio repeated for every song by that artist
    • âŒ Wastes storage space and causes update anomalies
  • No Artist or Album Entities: Everything crammed into MusicTrack
    • âŒ Can't query "all albums by artist X" efficiently
  • Playlist Design: Storing song_ids and song_titles as comma-separated lists
    • âŒ Violates first normal form; difficult to maintain

Separate entities: Artist, Album, Song, Genre, Playlist, plus junction tables like ArtistGenre, PlaylistSong
