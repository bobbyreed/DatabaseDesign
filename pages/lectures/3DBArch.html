Traditional File-Based Systems

Collection of application programs that perform services for end users.

Key Characteristics:

  • Each program defines and manages its own data
  • Data stored in separate files (CSV, XML, JSON)
  • No centralized control or management
  • Limited data sharing capabilities
Historical Context: Before databases, organizations relied on file-based systems, leading to numerous challenges we'll explore next.

Common File Formats

File-based systems use various formats for data storage:

CSV Example

student_id,name,email,gpa
1001,Bobby Reed,breed@okcu.edu,3.85
1002,Sarah Johnson,sjohnson@okcu.edu,3.92
1003,Mike Chen,mchen@okcu.edu,3.78

Characteristics

  • Simple text format
  • Comma-separated values
  • Limited data types
  • No relationships
  • No validation rules

Limitations of File-Based Approach (Part 1)

1. Separation and Isolation of Data

  • Each program maintains its own set of data
  • Users unaware of potentially useful data in other programs
  • No unified view of organizational data

2. Duplication of Data

  • Same data held by different programs
  • Wasted storage space
  • Potential for inconsistent values
  • Different formats for the same item

Limitations of File-Based Approach (Part 2)

3. Data Dependence

  • File structure defined in program code
  • Changes to file structure require program modifications
  • Maintenance becomes increasingly difficult

4. Incompatible File Formats

  • Programs written in different languages
  • Cannot easily access each other's files
  • Data exchange requires conversion programs

5. Fixed Queries/Proliferation of Programs

  • Programs written for specific functions
  • New requirements need new programs
  • Rapid growth in number of programs

The Database Approach

A revolutionary solution to file-based limitations:

Why It Arose:

  • Data definition embedded in applications ‚Üí need for separation
  • No control over data access ‚Üí need for centralized management
  • Data redundancy and inconsistency ‚Üí need for integration

The Solution:

Database + Database Management System (DBMS)

Key Insight: Separate data from applications, centralize control, and provide a unified interface.

Database Definition

A database is:

A shared collection of logically related data (and a description of this data), designed to meet the information needs of an organization.

Key Components:

  • Entities: Objects of interest
  • Attributes: Properties of entities
  • Relationships: Associations between entities
  • Metadata: Data about data

Characteristics:

  • Self-describing nature
  • Program-data independence
  • Support for multiple views
  • Concurrent data sharing

Database Management System (DBMS)

A software system that enables users to define, create, maintain, and control access to the database.

Database Application Programs:

Computer programs that interact with the database by issuing appropriate requests (SQL statements) to the DBMS.

Database Languages

Data Definition Language (DDL)

  • Permits specification of data types and structures
  • Defines constraints on data
  • All specifications stored in the database catalog
-- DDL Example
CREATE TABLE Student (
  student_id INT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(255) UNIQUE
);

Data Manipulation Language (DML)

  • General enquiry facility (query language)
  • SQL (Structured Query Language)
  • Insert, update, delete, and retrieve data
-- DML Example
SELECT name, email
FROM Student
WHERE student_id = 1001;

Understanding Architecture Levels

1. External Level (View Level)

  • Individual user views of the database
  • Different views for different user groups
  • Security through view restrictions
  • Customized data presentation

2. Conceptual Level (Logical Level)

  • Community view of entire database
  • Describes what data is stored
  • Relationships between data
  • Constraints and security rules

3. Internal Level (Physical Level)

  • Physical storage structures
  • File organizations and indexes
  • Storage allocation
  • Data compression and encryption

Database Views

Virtual tables based on database queries:

Benefits of Views:

  • ‚úì Reduce complexity for users
  • ‚úì Provide security through restricted access
  • ‚úì Customize data appearance per user needs
  • ‚úì Present consistent picture despite schema changes
-- Creating a view for students
CREATE VIEW ActiveStudents AS
SELECT student_id, name, email
FROM Students
WHERE status = 'Active'
  AND enrollment_date > '2024-01-01';
Design Principle: Views act as a window into your data, showing only what users need to see.

Evolution of Database Systems

First Generation (1960s-1970s)

  • Hierarchical Model: Tree-like structure (IBM's IMS)
  • Network Model: Graph structure (CODASYL)
  • Navigational access to data
  • Complex programming required

Second Generation (1970s-1990s)

  • Relational Model: Tables with rows and columns
  • SQL as standard query language
  • Data independence achieved
  • Examples: Oracle, DB2, SQL Server

Third Generation (1990s-Present)

  • Object-Relational: Objects + Relations
  • Object-Oriented: Pure OO databases
  • NoSQL: Document, Key-Value, Graph databases
  • NewSQL: Scalable SQL databases

  • SaaS: Software as a Service (Salesforce)

  • Database per service pattern
  • Event sourcing and CQRS
